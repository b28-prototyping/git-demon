## Context

The current road is an infinite flat plane — curvature only shifts scanlines horizontally. Classic pseudo-3D racers (OutRun, Lotus Turbo Challenge) achieve dramatic hills and valleys by defining the road as a sequence of Z-segments, each with a curvature (dx) and slope (dy) value. Segments are projected back-to-front with cumulative screen offsets, so a `dy > 0` segment pushes subsequent segments up on screen, creating a hill crest. This produces the iconic "seeing the road ahead rise and fall" effect.

## Implementation

### 1. Define `RoadSegment` struct

```rust
pub struct RoadSegment {
    pub z: f32,          // world-Z start of this segment
    pub curve: f32,      // horizontal curvature (existing curve_offset, but per-segment)
    pub slope: f32,      // vertical slope: +1 = uphill, -1 = downhill
    pub length: f32,     // Z-length of this segment
}
```

### 2. Build segment list in `WorldState`

Maintain a `Vec<RoadSegment>` representing the road ahead of the camera. Segments scroll toward the camera as `camera_z` advances. New segments are appended at the far end.

**Curvature source**: Current `curve_offset` / auto-steering sine waves become per-segment curve values, sampled at the Z where each segment was created.

**Slope source**: Introduce gentle sine-wave hills. Git activity bursts can trigger steeper hills/dips:
- Idle: `slope = sin(z * 0.001) * 0.3` — gentle rolling
- High activity: `slope = sin(z * 0.002) * 0.8` — dramatic crests

### 3. Project segments back-to-front

For each segment from far to near:
1. Project segment's Z through `Camera::project()` to get `(screen_y, depth_scale)`
2. Accumulate curve offset: `cum_dx += segment.curve * depth_scale²`
3. Accumulate slope offset: `cum_dy += segment.slope * depth_scale`
4. Final screen position: `(cx + cum_dx, screen_y + cum_dy)`

The cumulative offsets mean a hill at Z=2000 pushes everything behind it up, and the road reappears as it crests — exactly like OutRun.

### 4. Render road surface between segment boundaries

Each pair of adjacent projected segments defines a horizontal band. Fill the band with the grid texture (existing neon grid lines). The band's top/bottom edges may not be horizontal if slope differs, creating the visual of road surface pitching.

### 5. Update grid line rendering

Vertical grid lines now follow the cumulative curve offset per scanline row (they already do horizontally; now also vertically for hills). Horizontal grid lines are placed at segment boundaries rather than fixed Z intervals.

### 6. Sprite projection follows road segments

When projecting a sprite at `z_world`, find which segment it belongs to and apply the same cumulative `(dx, dy)` offsets. Sprites on a hill crest should rise with the road surface. This naturally emerges from using the segment offsets in `Camera::project()`.

### 7. Segment generation and recycling

- Keep ~40 segments ahead of camera (covering draw distance)
- Recycle segments that pass behind camera
- New segments sample from a curvature/slope noise function seeded by world time and git activity

## Acceptance Criteria

- [ ] Road visually rises and falls with hills
- [ ] Hills occlude the road and objects behind them (far segments hidden behind crests)
- [ ] Curvature still works and combines with hills (S-curves over hills)
- [ ] Grid lines follow the road surface through hills
- [ ] Sprites rise and fall with the road surface
- [ ] Hill intensity responds to git activity (more dramatic with higher CPM)
- [ ] No Z-fighting or visual tearing at segment boundaries
- [ ] Performance: segment projection adds < 0.5ms per frame at 200×60 terminal
